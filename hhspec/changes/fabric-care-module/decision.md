# 关键决策：fabric-care-module

## 决策 1：数据模型 - 结构化存储 vs 文本存储

**选择**：结构化存储（3 张新表）

**理由**：
1. 面料成分需要百分比校验（每层总和=100%），文本无法实现
2. 护理标签需要多语言 + 符号显示，结构化支持字典管理
3. 支持按面料材质筛选商品（未来功能扩展）
4. 消费端 PDP 需要分层展示，结构化数据易于渲染

**备选**：保留原型的文本框存储
- 被否决原因：无法实现百分比校验，护理标签无法标准化，用户体验差

## 决策 2：枚举设计 - IntEnum vs StrEnum

**选择**：IntEnum（FabricLayer / FabricMaterial 均存储整数）

**理由**：
1. 与项目现有枚举约定一致（已重构为 IntEnum）
2. 数据库检索效率高（int 索引优于 varchar）
3. 前端通过整数映射到枚举对象（intEnumOptions 工具函数）

**备选**：StrEnum 存储字符串
- 被否决原因：与项目重构方向冲突，检索效率低

## 决策 3：护理标签 - 预置数据 vs 用户自定义

**选择**：预置 9 条 ISO 3758 标准标签 + 支持系统管理员维护

**理由**：
1. 婚纱护理说明相对标准，ISO 3758 覆盖主要场景
2. 预置数据确保一致性和专业性
3. 保留字典维护接口，支持未来扩展新标签
4. CareInstructionDef 表设计 status 字段支持禁用/启用

**备选**：完全用户自定义
- 被否决原因：容易产生不规范的护理说明，用户体验不一致

## 决策 4：百分比校验 - 前端 vs 后端

**选择**：前端实时校验 + 后端强制校验（双重保障）

**理由**：
1. 前端实时校验提升用户体验（即时反馈）
2. 后端强制校验确保数据完整性（防止 API 直接调用绕过前端）
3. 后端使用 ProductUpsertValidator 集中校验逻辑

**备选**：仅后端校验
- 被否决原因：用户体验差，提交后才发现错误

## 决策 5：消费端展示 - 独立组件 vs 内联文本

**选择**：独立 ProductFabricCare 组件（可折叠）

**理由**：
1. 面料成分需要分层展示（Shell/Lining/Overlay），结构化组件易于维护
2. 护理标签需要显示 Unicode 符号 + 多语言文案
3. 移动端支持折叠，节省空间
4. 符合高端婚纱电商标准（参考 David's Bridal）

**备选**：纯文本显示
- 被否决原因：无法展示分层结构和符号，用户体验差
