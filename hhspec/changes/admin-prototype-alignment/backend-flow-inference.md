# 后端流程推断

> 本文件为 Phase 2.3 LLM 推断产物，作为 Phase 2.3.1 维度激活的信号源，不作为下游直接消费。
> 推断结论需经 Phase 2.3.1 用户确认后，沉淀到 decision.md 的「后端关键决策」章节。

## 识别到的业务模式

- 对齐型变更：前端 UI 回对原型为主，但 4 处缺口需后端数据/接口支撑
- 批量状态变更（Products 批量上架/下架/推荐/删除）
- 数据导出（Products 导出、Orders 导出订单）
- 列表 DTO 字段扩展（Orders 列表需地区 + 商品数）
- 品类属性配置入口重组（纯前端信息架构调整，复用既有 E-CAT-15~21 接口）

## 推断的后端机制

- **[事务]** Products 批量操作（上架/下架/推荐/删除）涉及多商品状态变更的原子性：全部成功或逐条容错（部分成功 + 失败清单）需要决策；批量删除还需复用单删的 409509（被订单引用不可删）校验
- **[幂等]** 批量操作重复提交（双击/网络重试）应保证幂等，状态置位类操作天然幂等，删除需容忍已删除
- **[权限]** 批量操作与导出沿用既有 admin RBAC 权限模型（catalog:write / order:read），无新增权限点；导出含客户邮箱等 PII，需确认导出权限是否收紧
- **[审计日志]** 批量操作与导出属于敏感批量动作，建议记录操作日志（已有 identity/audit 域 OperationLog 机制可复用）
- **[限流降级]** 导出全量订单/商品可能产生大查询；需限定导出范围（当前筛选条件 + 上限行数）避免拖垮 DB
- **[外部集成]** Refunds 行内拒绝仍走既有 Stripe 退款链路，无新集成；拒绝原因必填约束保持后端不变
- **[状态机并发]** 批量上架/下架与单品编辑并发时的状态覆盖：批量操作按当前状态做条件更新即可，无需引入版本号

## 待确认事项

1. 批量操作 API 形态：新增批量端点（一次请求 N 个 id）vs 前端循环调用单品端点 —— [事务][幂等]
2. 导出实现方式：前端本地序列化当前页/全部已筛选数据 vs 后端 CSV 导出端点 —— [限流降级][权限]
3. AdminOrderListItem 扩展「地区（国家）+ 商品数」：列表 SQL 需 join 地址快照与订单行计数，确认可接受的查询成本 —— [限流降级]
