# 关键决策：admin-prototype-alignment

> 变更类型：alignment（原型 = 验收基准）
> 来源：portal-api-integration 验收发现实现未对齐原型，经 16 页逐页对比审计（page_diff_summary）+ 用户逐项裁决产出本决策集。

## 决策 1：Categories 信息架构回对原型（推翻 PAGE-CAT-A04）
- **选择**：属性字典 + 品类×属性矩阵合并回 Categories 页作为第二个 Tab，恢复原型 3-Tab 结构（标准品类 / 属性集与字典 / 自定义标签）；废弃独立 AttributeSets 页（侧边栏菜单移除，路由重定向到 Categories 对应 Tab）；Tab 文案恢复「品类×属性矩阵」
- **理由**：用户验收以原型为基准，不接受拆页偏离
- **备选**：保留拆页（属性管理职责单一）——被否决，原型还原优先；保留双入口——被否决，避免重复入口

## 决策 2：Categories 品类卡片属性集徽章恢复点击交互
- **选择**：点击品类卡片上的属性集徽章直接打开该品类的属性三态配置抽屉（原型同款交互）
- **理由**：原型交互快捷顺手
- **备选**：徽章纯展示、配置统一走编辑抽屉——被否决

## 决策 3：补齐「子品类属性覆盖」只读概览卡片区
- **选择**：在合并后的矩阵 Tab 内补齐按根品类分组的子品类 delta 覆盖卡片区（只读，编辑入口仍在品类卡片）
- **理由**：原型有、实现完全缺失（真缺口）

## 决策 4：Products 补齐批量操作 / 导出 / 销量列 / 筛选
- **选择**：恢复勾选框列 + 批量操作栏（批量上架/下架/推荐/删除）、导出按钮、销量列、更多筛选面板的商品类型与标签筛选
- **理由**：原型功能缺失，中严重度

## 决策 5：Dashboard 补齐发布入口
- **选择**：PageHeader 恢复「发布站点」按钮（跳转 Publish 页）；快捷入口恢复 5 项（含编辑首页、发布站点）
- **理由**：原型功能缺失

## 决策 6：Orders 严格回对原型列
- **选择**：表格列「地区/商品数」替换「币种/承运」；补「导出订单」按钮
- **理由**：用户裁决严格对齐原型
- **备选**：双列并存——被否决；接受列调整只补导出——被否决

## 决策 7：Refunds 回对行内审批交互
- **选择**：恢复原型行内同意/拒绝按钮；点「拒绝」后按钮位置行内展开输入框填写原因（后端拒绝原因必填约束不变）
- **理由**：兼顾原型观感与后端必填约束
- **备选**：保留弹窗二次确认（资金操作更安全）——被否决；一键拒绝无原因——被否决（需放宽后端约束）

## 决策 8：低严重度显式偏离全部接受现状
- **选择**：维持实现现状：DEC-ANA-FE-2（KPI delta 行删除）、DEC-ANA-FE-3（待办瓦片 3 列）、DEC-ANA-FE-7（商品热度列改销售额）、FORM-REV-A01（Reviews Q&A 当前页过滤/chips 计数/商品图片）、DEC-SHP-FE-1/2（Shipping 即时持久化与行内编辑）
- **理由**：用户裁决未勾选回对，原设计决策保留

## 决策 9：反向缺口（实现多于原型）全部保留
- **选择**：三语支持（LocaleTabs）、CRUD 表单抽屉、删除二次确认、骨架屏/空态/分页/防抖搜索、退款工单表、定制尺码展开、SKU 并发版本控制等实现增强全部保留，不回退
- **理由**：真实业务能力，原型未画不代表不需要；回退是倒退

## 决策 10：L1 限界上下文与 domain
- **选择**：复用既有域不新建。domain_code = catalog（主）；trading（订单 DTO 扩展/导出/退款前端交互）；analytics/showroom 仅前端入口调整
- **理由**：纯对齐变更，无新业务边界
- **接口变更**：3 个新增端点（商品批量操作、商品 CSV 导出、订单 CSV 导出）+ AdminOrderListItem 扩展 country/itemCount，均非 breaking
- **DB schema**：零变更（country 取地址快照、itemCount 行计数，查询层解决）
- **错误/认证策略**：沿用现有（409xxx/422xxx 错误码体系、admin JWT + RBAC，不新增权限点）

## 后端关键决策

> 来源：Phase 2.3.1 后端实现深度探索
> 下游消费：L0 flow_modeler（已映射到 business-flow.yml 节点）、L1 architect（映射到 api-contracts / error-strategy）

### BE-DIM-4 状态机/并发/事务
- **决策**：Products 批量操作新增后端批量端点（一次请求 N 个 id），逐条容错执行，返回成功/失败清单（删除复用 409509 被订单引用校验），整体不回滚；操作记录审计日志
- **触发信号**：backend_inference[事务]
- **理由**：避免前端 N 次请求；逐条容错语义对运营更友好（部分成功可见）
- **约束**：L1 API 契约须定义批量响应结构（successIds / failures[{id, errorCode, message}]）；批量操作天然幂等（状态置位/删除容忍已删除）

### BE-DIM-8 性能与可扩展性
- **决策**：Products/Orders 导出采用后端 CSV 导出端点，按当前筛选条件全量导出，上限 10000 行（超出截断并在响应头/尾行提示），记录审计日志
- **触发信号**：backend_inference[限流降级]
- **理由**：数据完整且可控；PII（客户邮箱）导出可审计
- **约束**：L1 契约须定义导出端点的筛选参数与上限行为；导出查询须走分页流式读取避免大结果集内存压力

- **决策**：AdminOrderListItem DTO 扩展 country（地址快照提取）+ itemCount（订单行计数）两字段，列表 SQL 以 join/子查询提供
- **触发信号**：backend_inference[限流降级]
- **理由**：避免前端逐行 N+1 取详情
- **约束**：L2 数据设计须验证列表查询计划（行计数子查询需命中 order_id 索引）；历史地址快照均含国家字段（前提假设④），无迁移

## 原型强对照约束

> 触发：linked_prototype_snapshots 非空（16 页）且 prototype_dir 有值；alignment 类型默认启用。

1. **原型 = 验收基准**：本次为纯对齐变更，原型不再修改（前提假设①）。L4 验收以 `hhspec/prototype/portal-admin/src/views/*.vue` 16 页为 ground truth 做全量对照。
2. **样式 token**：原型 `tailwind.config.js` 自定义 token（gold/ink/canvas/line/rounded-luxe 等）为权威，实现端 token 不得偏离精确值。
3. **proto_is_frontend = true**：原型即 Vue SFC，修复时优先「复制 + 适配」原型对应区块（保留实现端已接入的 store/API 数据层与三语等增强，仅回对模板结构与交互）。
4. **L2/L3 约束**：L2 前端详设须对每个修复页面产出与原型逐区块对照的组件方案；L3 实现禁止自由发挥布局。
5. **L4 约束**：UI 验收四维对照（结构/内容/交互路径/样式 token）覆盖本次 16 页中被修改的页面；决策 8 接受的偏离项列入豁免清单不计差异。
